# Phase 5 Task Studio V1 scope pack
Version: 1.0
Status: Accepted
Task: P5.6 — Task Studio V1 scope pack
Artifact ID: reuse.phase5-task-studio-v1-scope-pack.v1
Reuse scope: Accepted Task Studio V1 scope pack covering initial task-family exposure, surface exposure, result/proof expectations, non-goals, initial build scope, and onboarding/category-teaching constraints over accepted Task Studio semantics

This artifact is accepted for downstream use.
It narrows V1 exposure and defaults only.
It does not reopen accepted commissioned-work semantics, Task Studio lifecycle meaning, or the R7 handoff payload.

## 1. Purpose

This pack turns the accepted Task Studio app-model contract into one conservative V1 exposure boundary.

It exists to:

- make the V1 task-family and surface scope explicit,
- lock the minimum result, proof, delta, and writeback expectations that V1 may not collapse,
- constrain onboarding and category teaching so V1 remains run-native and commissioned-work-first,
- refuse overbuild into builders, marketplaces, secret management, or app-private execution semantics.

## 2. Scope boundaries

### In scope

- the exact V1 task-family set,
- the V1 primary and companion Task Studio surfaces,
- V1 floors for result, proof, delta, review, and writeback visibility,
- conservative V1 defaults for reusable-execution lineage exposure,
- onboarding and category-teaching constraints,
- explicit non-goals that protect against uncontrolled sprawl.

### Out of scope

- changing the accepted lifecycle loop, task-family mapping, or progressive-disclosure rules,
- changing run-class, protocol, governance, proof, or handoff semantics,
- route/module/component implementation details that belong to `P6.1`,
- trigger builders, workflow designers, applet builders, pack stores, or integration-secret management UX,
- domain-specific surfaces or code-domain-first specialization.

## 3. V1 interpretation rules

### 3.1 Exposure-not-semantics rule

`P5.6` narrows initial Task Studio exposure, default emphasis, and onboarding sequence only.
It may not introduce a new task ontology, a new lifecycle, route-local truth, or app-private execution objects.

### 3.2 Exact-seven-family rule

V1 includes exactly the accepted seven Task Studio families: `Compare`, `Synthesize`, `Extract`, `Transform`, `Audit`, `Plan`, and `Triage`.
They remain direct projections over the accepted base run classes `compare`, `synthesize`, `extract`, `transform`, `audit`, `plan`, and `triage`.
Onboarding may order or explain these families, but it may not rename, merge, hide, or replace them with persona, mode, or agent-builder taxonomies.

### 3.3 Stage-centered floor rule

V1 keeps the full commissioned-work loop explicit: `commission -> interpret -> lock -> execute -> inspect -> accept/branch -> persist selectively`.
Each stage keeps at least one named Task Studio surface center, and consequential work may not fall back to `Task Home`, `Console`, or chat-style answer blobs as the only truth surface.

### 3.4 Governable-output floor rule

Every consequential V1 run must preserve typed result state, `Proof Bundle`, `State Delta`, and linked review/apply posture where relevant.
Default views may summarize, but proof, delta, and authority consequences remain one step away and must stay inspectable for success, no-op, blocked, and failure outcomes.

### 3.5 Conservative persistence-and-reuse rule

V1 may expose reusable-execution lineage and pinned refs wherever they materially govern a run, but it does not promise full trigger/applet/pack/binding authoring UX.
Artifact-lane review/apply is the default V1 persistence happy path; other lanes remain explicit, lane-separated, and typically inspectable or deferred rather than hidden or collapsed.

### 3.6 Onboarding-is-pedagogy rule

V1 onboarding teaches commissioned-work semantics; it does not define them.
Teaching may simplify language, but it must preserve the distinction between objective, evidence, context, authority, proof, delta, and writeback.

## 4. V1 task-family scope

| Task Studio family | Accepted run class | V1 posture | V1 output expectation |
| --- | --- | --- | --- |
| `Compare` | `compare` | Included in V1 | Explicit compared targets, structured comparison result, and proof basis for recommendation or divergence |
| `Synthesize` | `synthesize` | Included in V1 | Bounded result or artifact draft with claim-to-evidence proof and uncertainty labeling |
| `Extract` | `extract` | Included in V1 | Structured captured fields or spans with field-to-source proof trace |
| `Transform` | `transform` | Included in V1 | Targeted artifact change with explicit delta, writeback proposal posture, and compensation visibility |
| `Audit` | `audit` | Included in V1 | Criteria-based evaluation with evaluator results, failure basis, and review-ready judgment framing |
| `Plan` | `plan` | Included in V1 | Options, checkpoints, or branch recommendations with explicit assumptions and comparison basis |
| `Triage` | `triage` | Included in V1 | Intake/routing result that makes commission/contract narrowing explicit instead of leaving it in bridge-private state |

V1 adds no eighth family, no domain-private subfamily, and no Task-Studio-only taxonomy.
If onboarding uses starter examples or preferred ordering, that ordering remains pedagogical and reversible.

## 5. V1 surface scope

### 5.1 Named Task Studio surfaces in V1

| Surface | Lifecycle role | V1 posture | Exposure rule |
| --- | --- | --- | --- |
| `Task Home` | `commission` index and resume center | Included as a primary entry/resume surface | May aggregate resumable work, but may not replace stage-specific truth surfaces |
| `New Commission Sheet` | `commission` intake shaping | Included as a primary entry surface | Must create explicit draft `Commission` and source grounding rather than executing serious work directly |
| `Task Contract Panel` | `interpret` / `lock` contract shaping | Included as a primary consequential surface | Must show run class, deliverable shape, acceptance policy, and protocol pins in structured form |
| `Evidence Pack Builder` | `interpret` evidence admission | Included as a primary consequential surface | Must preserve inclusion/exclusion, pinning, and source-region lineage |
| `Context Inspector` | `interpret` / `lock` compiled basis inspection | Included as a primary consequential surface | Must expose compiled context, frozen admitted basis, and lineage without hiding them in `Console` |
| `Strategy Board` | `interpret` / `accept/branch` planning and branching | Included as a companion-first surface | Appears when option comparison, replay, or branching materially matters; it is not a separate task ontology |
| `Authority Panel` | `lock` authority and review path | Included as a primary consequential surface | Must expose authority scope, review path, side-effect preview, and writeback preview before execution or apply |
| `Live Run View` | `execute` execution continuity | Included as a primary consequential surface | Must show run state, blocked reasons, approvals, child-run lineage, and reusable-execution refs |
| `Result Canvas` | `inspect` / `accept/branch` focal result | Included as a primary consequential surface | May foreground result state, but must link to proof, delta, and next consequential actions |
| `Proof Ledger` | `inspect` / `persist selectively` proof and approval history | Included as a required companion surface | Must remain directly reachable in all consequential outcomes |
| `Delta Inspector` | `inspect` consequence state | Included as a required companion surface | Must expose explicit change state, including empty or failure deltas |
| `Writeback Panel` | `persist selectively` selective persistence | Included as a primary consequential surface | Must keep lane-local proposal, approval, apply status, and compensation lineage explicit |

### 5.2 Companion grammar exposure in V1

`Browser / Library`, `Queue / Inbox`, `Board`, `Compare View`, `Timeline`, `Diff / Merge View`, and `Console` remain valid V1 companions because they are part of the accepted shared grammar.
V1 does not promote them into replacement truth models for contract, authority, proof, delta, or writeback semantics.

## 6. V1 result, proof, delta, and writeback floor

| Semantic area | V1 floor | Explicit refusal |
| --- | --- | --- |
| Result | Every consequential run returns a typed result shaped by the contracted run class or an explicit blocked/failure terminal state | No generic answer blob may replace structured result state |
| Proof | `Proof Bundle`, evaluator outcomes, uncertainty, omissions, contradictions, and failure basis remain inspectable one step away from the default view | No proof-only-in-logs or prose-only confidence claims |
| Delta | `State Delta` remains explicit for success, no-op, blocked, and failure outcomes | No silent mutation and no disappearance of consequence state on failure |
| Review / approval | Review and approval state remain attributable and append-only wherever proof or writeback matters | No bridge-private verdict that replaces `Review` or `Approval Decision` |
| Writeback | Artifact-lane review/apply is the default V1 persistence path; other lanes remain explicit, lane-separated, and inspectable/deferred | No mixed-lane convenience, no silent auto-apply, and no hidden canon or workflow promotion |
| Failure / replay | Failed, blocked, and resumed runs stay inspectable with proof/delta lineage and replay/branch continuity | No generic retry button that erases basis, branch options, or residual uncertainty |

V1 therefore keeps proof-returning commissioned work real even when the initial UX stays conservative.

## 7. V1 onboarding and category-teaching constraints

1. Onboarding teaches the commissioned-work loop, not a chat prompt ritual.
2. The initial teaching path must preserve the distinction between `Evidence Pack` and `Context Pack`, between `Authority Scope` and strategy, and between `Proof Bundle`, `State Delta`, and `Writeback Proposal`.
3. Task families are taught as projections over accepted run classes rather than as personas, assistants, or opaque agent modes.
4. Teaching copy may introduce recommended starter paths, but it may not hide the accepted family vocabulary once work becomes consequential.
5. No onboarding step may reduce authority, review, proof, or writeback to a one-click confirmation pattern.
6. No onboarding or category label may imply that Task Studio is code-editing-first, transcript-first, or agent-builder-first.

## 8. V1 reusable-execution exposure constraints

| Object family | V1 expectation | V1 non-goal |
| --- | --- | --- |
| `Protocol` | Show `protocol_id`, `protocol_version`, run class, and proof/review posture where materially relevant | No protocol authoring or floating-version selection UI |
| `Workflow` | Surface pinned `workflow_id` and stage lineage when a run is workflow-backed | No workflow builder or stage-editor UX |
| `Trigger` | Keep `trigger_id` and trigger provenance inspectable when runs arrive from scheduled/event launches | No trigger-creation, enable/disable, or policy-management UX |
| `Applet` | Keep `applet_id` and launch context inspectable when reusable execution hands into Task Studio | No applet builder, marketplace, or hidden autonomy widening |
| `Pack / Extension` | Keep `pack_id` and member provenance inspectable when packaged behavior matters | No pack store, pack authoring, or pack lifecycle management surface |
| `Integration Binding` | Keep `integration_binding_ref` and scoped capability context inspectable in authority/run/apply surfaces | No inline secret entry, rotation manager, or binding-ownership rewrite |

Background and resumed runs keep the same governance and proof floor as interactive runs.
Reusable-execution lineage remains inspectable in V1; authoring and administration remain deferred.

## 10. Boundary locks for downstream work

1. Human review may change default teaching emphasis or first-run ordering, but it may not add new task families, remove accepted named surfaces, or redefine progressive disclosure.
2. `P5.7` must preserve the full R7 handoff payload even when V1 default landings prefer `Task Home` or other stage-specific Task Studio surfaces.
3. `P6.1` must map route, module, and component contracts onto this V1 exposure floor rather than reopening Task Studio semantics.
4. V1 refuses builder, marketplace, or secret-management overbuild for Protocol, Workflow, Trigger, Applet, Pack, or Integration Binding objects.
5. Non-artifact write lanes may be deferred from the V1 happy path, but they must remain explicit, lane-separated, and recoverable from the same shared object model.

## 11. Downstream implications

### 11.1 For `P5.7`

- the commissioning-handoff contract should use this pack as the accepted V1 exposure floor for default landing surfaces, primary/companion splits, and artifact-lane-first persistence,
- the handoff pack may narrow defaults further only when it preserves the full payload and makes any fallback routing explicit.

### 11.2 For `P6.1`

- surface-contract work should treat these named surfaces and companion grammars as the V1 implementation target,
- route/module/component planning may vary presentation, but it should not collapse authority, proof, delta, or writeback into secondary or hidden status.

## 12. Review notes

Human review should confirm that this pack:

- narrows V1 exposure without reopening accepted Task Studio or R7 semantics,
- keeps the accepted seven task families intact while refusing taxonomy sprawl,
- preserves explicit proof, delta, review, and writeback visibility in the initial build,
- keeps reusable-execution lineage inspectable without overbuilding authoring or secret-management surfaces,
- treats onboarding and category teaching as pedagogy rather than semantic canon.

Human review should also explicitly confirm the preferred default for:

- whether R7 handoff lands first in `Task Home` or the run's current lifecycle surface,
- whether artifact-lane apply is in the initial V1 happy path or begins as inspect/review-first,
- whether any starter ordering of the seven families is used in onboarding.
