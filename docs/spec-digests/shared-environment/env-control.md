### 10. E. REUSE AND COMPOSITION PRIMITIVES
#### E1. Template

Primitive type:
Environment-native

Definition:
A reusable partially-instantiated setup for recurring work.

Examples:
- standard compare setup
- memo draft setup
- vendor assessment setup
- issue triage setup

Minimum required fields:
- template_id
- base contract or form schema
- default attachments or source slots
- default output schema
- scope
- owner
- version

Required behaviors:
- instantiate
- clone
- version
- share
- deprecate

Important boundary:
A Template is useful but shallow.
It is not the full successor to skills.md.

#### E2. Protocol

Primitive type:
Environment-native and critical

Definition:
A versioned executable specification for a class of work.

Why it is primitive:
This is the real successor to unstructured skills.md files.
Without Protocol, reusable work falls back into prose instructions and weak prompt wrappers.

Minimum required fields:
- protocol_id
- name
- version
- run class
- parameter schema
- contract schema
- evidence rules
- context recipe bindings
- strategy preset bindings
- authority defaults
- verifier pack bindings
- proof policy
- writeback defaults
- failure classes
- escalation rules
- supported projections

Required behaviors:
- instantiate runs
- validate inputs
- evolve versions
- attach tests or fixtures
- distribute inside Packs
- wrap inside Applets

Important constraint:
A Protocol is not just a prompt template.
It is an executable work specification.

#### E3. Context Recipe

Primitive type:
Environment-native

Definition:
A reusable policy for how Context Packs should be compiled.

Minimum required fields:
- context_recipe_id
- target run classes
- inclusion rules
- exclusion rules
- retrieval rules
- freshness rules
- trust rules
- memory injection rules
- canon injection rules
- compression policy
- freeze policy

Required behaviors:
- bind to protocols
- bind to templates
- bind to workflows
- compare recipe variants
- explain why context was assembled a certain way
- materialize explicit memory/canon admission inputs for admitted runs

Important boundary:
A Context Recipe is a reusable policy primitive.
It may guide environment-control admission for memory/canon lane inputs, but it is not itself the admitted per-run lane input.

#### E6. Workflow

Primitive type:
Environment-native composition primitive

Definition:
A composed graph of runs, conditions, approvals, triggers, and writebacks.

Why it is primitive:
The environment must support recurring or multi-step work, but it should compose from runs rather than replacing them.

Minimum required fields:
- workflow_id
- name
- version
- steps
- conditions
- triggers
- checkpoints
- approvals
- writeback destinations
- scope
- owner
- installed packs or dependencies

Required behaviors:
- run
- pause
- resume
- cancel
- version
- inspect state
- route assignments
- branch
- replay

Important constraint:
Workflow is composed from the run model.
It is not allowed to invent a separate execution ontology.

#### E8. Applet

Primitive type:
Environment-native and critical

Definition:
A packaged user-facing projection over one or more Protocols, with state, surfaces, roles, triggers, and writeback behavior.

Why it is primitive:
This is how tasks become quasi-apps.
Without Applet, reusable work remains a saved task rather than a real environment-native mini-application.

Minimum required fields:
- applet_id
- name
- version
- protocol bindings
- surface bindings
- local state model
- supported scopes
- role model
- trigger bindings
- writeback behavior
- integration requirements

Required behaviors:
- install
- open as an app-like surface
- accept inputs
- instantiate runs or workflows
- route review and approvals
- save state
- evolve version

Important constraint:
An Applet is not merely a saved workflow.
It is a reusable, user-facing, scoped work application inside the environment.

### 12. G. PROJECTION GRAMMAR PRIMITIVES
These are reusable environment projection primitives.
They are not the same as semantic objects like Run or Artifact.
They are reusable surface grammars that every app builder can use so the environment stays coherent.

#### G1. Composer

Definition:
The standard projection for entering a Commission, filling parameters, attaching sources, or instantiating a Template, Protocol, or Applet.

#### G2. Canvas

Definition:
The primary work or result surface for a run, artifact, or applet.

#### G3. Inspector

Definition:
The detail rail or detail surface for Contract, Evidence Pack, Context Pack, Authority Scope, Proof Bundle, State Delta, or metadata.

#### G4. Browser / Library

Definition:
The standard projection for listing and filtering Runs, Artifacts, Memory Objects, Canon Objects, Templates, Protocols, Applets, or Packs.

#### G5. Timeline

Definition:
The lifecycle projection for run history, checkpoints, branch points, approvals, replays, and major events.

#### G6. Ledger

Definition:
The structured projection for proof, validations, decisions, writeback lanes, and audit trails.

#### G7. Compare View

Definition:
The standard projection for comparing branches, evidence packs, strategy presets, artifacts, proofs, deltas, or versions.

#### G8. Queue / Inbox

Definition:
The standard projection for Assignments, pending Reviews, pending Approvals, failed runs, escalations, and unresolved governance actions.

#### G9. Board

Definition:
The grouped progression surface for staged work, multi-run processes, or workflow state clusters.

#### G10. Diff / Merge View

Definition:
The standard projection for before/after inspection and selective acceptance of changes.

#### G11. Live Monitor

Definition:
The standard active-execution projection for a running Run or Workflow.

#### G12. Console

Definition:
The lower-level diagnostic projection for detailed logs, tool outputs, test results, runtime traces, or advanced inspection.

Important rule:
Apps may style these projections differently, but they should not invent completely incompatible equivalents unless a domain genuinely requires it.
