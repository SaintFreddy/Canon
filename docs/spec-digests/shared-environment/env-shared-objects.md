## Source: All-Prims — Shared Environment Purpose and Design Requirements

### Shared Environment
This document defines the primitive set for the Shared Environment layer.

It does not redefine the Agentic Infrastructure Engine primitive set.
It does not define the primitive set for any single app or wedge.
It does not define only the MVP.

It defines the environment-level primitives that sit above the engine and below individual apps.

These primitives exist so that:
- multiple first-party apps can live on one coherent substrate
- third-party builders can extend the environment without inventing incompatible semantics
- runs, memory, canon, artifacts, branches, approvals, and workflows can move across apps without collapsing into copy-paste blobs
- the environment remains a real platform rather than a bundle of wrappers

These primitives are therefore:

- broader than any one app
- narrower than the universal engine SDK
- reusable across the whole environment
- intended to be open to builders if the environment itself is open and extensible

In one sentence:

The engine defines what serious agentic systems can do.
The environment defines how serious agentic work lives, is governed, is reused, and is handed off across apps.
## 1. EXECUTIVE DEFINITION
### Engine
The Agentic Infrastructure Engine is a headless execution substrate and developer SDK for building governed agentic systems.

It is not merely:
- an LLM wrapper
- a tool-calling helper
- a planner
- a vector database abstraction
- a workflow engine
- a memory layer
- a multi-agent framework

It includes all of those categories, but none of them alone is sufficient.

The engine exists to provide a coherent set of first-class primitives for:
- bounded execution
- context compilation
- evidence handling
- source retrieval
- memory
- accepted knowledge
- planning and orchestration
- tool and model interaction
- validation and proof
- authority and side-effect control
- persistence and state deltas
- provenance and replay
- observability and governance hooks
### Shared Environment
The Shared Environment is the human-facing operating layer built on top of the Agentic Infrastructure Engine.

Its job is not to replace the engine.
Its job is to standardize the shared object model, interaction grammar, governance model, and reuse model for all apps inside the environment.

Without a distinct environment primitive set, each app will reinvent:
- scope
- projects
- runs
- review flows
- writeback logic
- branching
- memory participation
- canon participation
- artifacts
- approvals
- workflows
- packaging

At that point the “shared environment” would be fake.
It would merely be a shell around unrelated products.

The environment primitive set exists to prevent that.

### 2. ARCHITECTURAL STANCE
The engine should be universal, but not undefined.

It should be model-agnostic, but not lowest-common-denominator.
It should be modular, but not fragmented.
It should be extensible, but not vague.
It should be capable of powering simple and complex systems with the same underlying primitives.

The correct stance is:

- minimal but real core contracts
- pluggable policies and adapters
- typed first-class objects
- evented lifecycle
- explicit side effects
- provenance by default
- replayability by design
- compiled context as a universal primitive
### 3. OPINIONATED UNIVERSALS THAT SHOULD BELONG IN THE ENGINE
The engine should avoid product-specific opinions, but there are certain opinions that should be treated as universal because otherwise the SDK becomes structurally weak.

#### 3.1 Bounded execution units

Even if a host product does not expose the concept publicly, the engine needs an internal bounded execution object.

Without that, the engine cannot support:
- orchestration
- attribution
- replay
- cancellation
- checkpointing
- cost tracking
- side-effect review
- proof bundling
- failure recovery
- provenance

The host application may name this however it wants:
- run
- job
- operation
- session
- procedure
- execution
- task instance

The engine should remain neutral on naming, but it should not abandon the primitive.

#### 3.2 Compiled context

Context should be compiled rather than implicitly inherited from transcript history or ambient application state.

This should be treated as universal.

The engine should support:
- automatic context compilation
- assisted compilation
- manual control
- frozen compiled context
- context diffs
- inclusion and exclusion reasoning
- summarized versus verbatim distinctions

#### 3.3 Evidence as a distinct class from generic context

The engine should distinguish evidence-bearing sources from general runtime context.

This distinction matters for:
- verification
- citation
- source coverage
- claim support
- auditability

#### 3.4 Explicit authority boundaries

The engine should distinguish:
- what the system can infer
from
- what the system is allowed to do

Interpretation and permission are not the same thing.

#### 3.5 Persistence lanes

The engine should distinguish:
- output generation
- durable storage
- memory promotion
- accepted-knowledge promotion
- external side effects

These should not collapse into one muddy write action.

#### 3.6 Provenance by default

The engine should assume that source lineage, transformation lineage, and decision lineage are first-class requirements, not optional extras.

#### 3.7 Replayability

The engine should support reconstruction, comparison, and replay as native capabilities rather than afterthoughts.

#### 3.8 Verification as a core layer

The engine should not assume that generation is sufficient.
It should treat validation, checking, and proof assembly as first-class.

These are opinions, but they are good universal opinions.
### 4. WHAT THE ENGINE MUST NOT ASSUME
The engine must not assume:
- a chat UX
- a transcript-centric product
- a visual workflow builder
- a specific planning strategy
- a specific memory model implementation
- a specific storage backend
- a single model provider
- a single tool protocol
- a single domain ontology
- a single approval model
- a single truth model for all applications
- that all host applications expose the same nouns to end users

The engine should define primitives, lifecycle contracts, and extension points.
It should not dictate a consumer product.

## PART II
### SHARED ENVIRONMENT — PRIMITIVE SET
### 3. TEST FOR WHETHER SOMETHING BELONGS IN THE ENVIRONMENT PRIMITIVE SET
A thing belongs in the environment primitive set if most of the following are true:

- it must be reusable across more than one app
- it must have a stable identity across app boundaries
- it must be scopable
- it must be permissionable
- it must be reviewable
- it must participate in provenance
- it must participate in branching or replay directly or indirectly
- it must participate in persistence or writeback decisions
- third-party builders would need it to build serious apps inside the environment
- if left undefined, multiple apps would likely invent incompatible versions of it

If an object must work in any host application anywhere, regardless of whether that application uses the Shared Environment, it belongs in the engine.

If an object only matters to one wedge or one domain pack, it belongs in that app or domain pack.
### 4. CORE DESIGN REQUIREMENTS
The environment primitive set must satisfy the following requirements.

#### 4.1 Cross-app continuity

Objects must preserve identity across apps.
A result from one app must not become an unstructured paste blob in another.

#### 4.2 Run-centered consequential work

The environment may support threads, chat, and continuity lines, but serious work must still resolve into bounded runs.

#### 4.3 Explicit scope

Every consequential object must declare where it lives and who can see or act on it.

#### 4.4 Explicit durability lanes

Results, memory, canon, workflows, and external actions must remain distinct lanes.
The environment must not collapse them into one generic save action.

#### 4.5 Real branching

Branches must fork more than transcripts.
They must be able to fork assumptions, context, strategy, candidate knowledge, artifacts, and outcomes.

#### 4.6 Governable reuse

Saved work must evolve from templates to protocols to applets to packs in a controlled way.

#### 4.7 Projection consistency

Different apps may look different, but they must share enough projection grammar that users do not have to relearn the platform from scratch in every app.

#### 4.8 Human-legible semantics

The environment exists to expose meaningful semantics to the user.
It should not reduce everything back to hidden prompts and opaque model behavior.
### 5. PRIMITIVE FAMILIES
The environment primitive set is divided into seven families:

A. Topology and Scope
B. Identity and Participation
C. Delegation and Execution
D. Grounding and Durable Knowledge
E. Reuse and Composition
F. Collaboration and Governance
G. Projection Grammar

The semantic primitive set consists mainly of A through F.
G provides reusable projection primitives so the whole environment shares one interaction grammar.
