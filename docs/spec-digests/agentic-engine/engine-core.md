## 0. DOCUMENT PURPOSE
### Engine
This document defines the Agentic Infrastructure Engine as an engine and SDK only.

It deliberately excludes product-shell and app-specific objects such as:
- app IDs
- workspace shells
- thread UIs
- chat metaphors
- environment-specific review surfaces
- product navigation structures
- domain-specific business ontologies

This document is not a description of the shared environment.
This document is not a description of any single host application.
This document is not a UX spec.

It is the end-to-end feature set for a universal, model-agnostic, source-aware, memory-capable, policy-aware, replayable SDK for agentic systems.

The core requirement is this:

A developer should be able to use this SDK to build:
- source-driven agentic orchestration
- memory-enabled applications
- retrieval-heavy research systems
- multi-agent execution systems
- verification-heavy automation
- approval-gated workflows
- replayable and auditable agent runtimes
- domain-specific products built on shared primitives

without being forced into a specific app model.
### Shared Environment
This document defines the primitive set for the Shared Environment layer.

It does not redefine the Agentic Infrastructure Engine primitive set.
It does not define the primitive set for any single app or wedge.
It does not define only the MVP.

It defines the environment-level primitives that sit above the engine and below individual apps.

These primitives exist so that:
- multiple first-party apps can live on one coherent substrate
- third-party builders can extend the environment without inventing incompatible semantics
- runs, memory, canon, artifacts, branches, approvals, and workflows can move across apps without collapsing into copy-paste blobs
- the environment remains a real platform rather than a bundle of wrappers

These primitives are therefore:

- broader than any one app
- narrower than the universal engine SDK
- reusable across the whole environment
- intended to be open to builders if the environment itself is open and extensible

In one sentence:

The engine defines what serious agentic systems can do.
The environment defines how serious agentic work lives, is governed, is reused, and is handed off across apps.
## 1. EXECUTIVE DEFINITION
### Engine
The Agentic Infrastructure Engine is a headless execution substrate and developer SDK for building governed agentic systems.

It is not merely:
- an LLM wrapper
- a tool-calling helper
- a planner
- a vector database abstraction
- a workflow engine
- a memory layer
- a multi-agent framework

It includes all of those categories, but none of them alone is sufficient.

The engine exists to provide a coherent set of first-class primitives for:
- bounded execution
- context compilation
- evidence handling
- source retrieval
- memory
- accepted knowledge
- planning and orchestration
- tool and model interaction
- validation and proof
- authority and side-effect control
- persistence and state deltas
- provenance and replay
- observability and governance hooks
### Shared Environment
The Shared Environment is the human-facing operating layer built on top of the Agentic Infrastructure Engine.

Its job is not to replace the engine.
Its job is to standardize the shared object model, interaction grammar, governance model, and reuse model for all apps inside the environment.

Without a distinct environment primitive set, each app will reinvent:
- scope
- projects
- runs
- review flows
- writeback logic
- branching
- memory participation
- canon participation
- artifacts
- approvals
- workflows
- packaging

At that point the “shared environment” would be fake.
It would merely be a shell around unrelated products.

The environment primitive set exists to prevent that.
## 2. LAYER BOUNDARY
The architecture has three layers.

#### 2.1 Engine layer

This is the universal SDK and execution substrate.
It includes engine-level primitives such as:
- execution units
- compiled context
- evidence
- memory
- accepted knowledge
- strategy
- tool invocation
- validation
- proof
- persistence proposals
- provenance
- replay
- policies

Those are universal, host-application-neutral primitives.

#### 2.2 Environment layer

This is the platform layer defined by this document.
It provides:
- the shared operating model
- the shared object model for apps
- the shared reuse model
- the shared governance model
- the shared packaging model
- the shared projection grammar
- the shared collaboration grammar

#### 2.3 App / domain layer

These are app-specific or domain-specific packs built inside the environment.
Examples:
- Task Studio
- repo-aware code editing
- research synthesis
- document/spec systems
- automation apps
- domain-specific review systems

The environment primitive set must be stable enough that each app can project its own UX without inventing a new ontology.

## Source: All-Prims — What Does Not Belong in the Engine

### 31. WHAT DOES NOT BELONG IN THE ENGINE CORE
The following should not be hard-coded into the core:

- a single consumer product vocabulary
- a specific UI review surface
- a chat transcript model
- a specific approval UX
- a specific memory retrieval algorithm
- a specific planning strategy
- a specific multi-agent topology
- a specific storage engine
- a single proof style for every task type
- a domain-specific schema that all adopters must inherit
- a single host application object model
- decorative agent visual metaphors

The core should define primitives and contracts.
Opinionated reference modules can live above that.
